# Cinemax - Jellyfin Client for Apple Platforms

Native Jellyfin client for iOS 26+ and tvOS 26+. "Cinema Glass" design system (dark glassmorphism, editorial layouts, no borders). SwiftUI multi-platform, single Xcode project (iOS + tvOS targets). Swift 6 strict concurrency.

> This file = rules, gotchas, and non-derivable context. Feature *behavior* is derivable from the code — read the owning file. Lines tagged **RULE** override default behavior.

## Architecture

- **CinemaxKit** local Swift Package at `Packages/CinemaxKit` — networking, models, persistence.
- `@Observable` + `@MainActor` for all state. `JellyfinClient` wrapped with `NSLock` + `nonisolated(unsafe)` for Sendable.
- **RULE — iOS `NavigationStack`**: destinations pushed via `navigationDestination(item:)` render in a separate context — `@Observable` changes won't re-render unless the destination is a standalone `View` struct with its own `@Environment` properties, not an extension method returning `some View`.
- **RULE — Lazy-container navigation**: SwiftUI silently ignores `navigationDestination(item:)` inside `LazyVGrid/LazyVStack/LazyHStack/List`. Hoist the modifier to a non-lazy ancestor; bubble the action up via a callback that mutates a screen-level `@State`. Reference: `AdminItemMenu.onSelectDestination` → screen-level `@State AdminMenuPushIntent?`. Used by `MovieLibraryScreen`, `MediaDetailScreen`.
- **RULE — Singleton root stores** (`AppNavigation.swift`): `AppState`, `NetworkMonitor`, `MenuConfigStore` are `private static let` process singletons assigned into `@State` — scene-event struct recreation could otherwise redo main-thread disk work. `ThemeManager`/`ToastCenter` are NOT in this set.
- **RULE — Hand-built URLs must preserve the server base path** (`URLComponents+ServerPath.swift`): assigning `components.path` directly REPLACES the whole path, silently dropping the sub-path of sub-path-hosted servers (`https://host/jellyfin`) — every hand-built URL 404s while SDK-routed calls keep working. Always route through `setEndpointPath(_:preservingBasePathOf:)`. Applies to all hand-built endpoints (playback direct-stream, every `ImageURLBuilder` URL, SyncPlay REST/socket); Widget + Top Shelf carry a local copy (can't link CinemaxKit).

### iOS 26 / tvOS 26 API rules (all RULE)

- **`UIButton`**: use `UIButton.Configuration`; never `UIButton(type:)` + `setTitle/titleLabel?.font/backgroundColor/contentEdgeInsets`. Pattern in `NativeVideoPresenter`. Frosted bg via `config.background.customView = UIVisualEffectView(...)`.
- Free SwiftUI helpers returning `some View` that touch `PrimitiveButtonStyle.plain`/`Font`/etc. must be `@MainActor`.
- **iPad**: `UIRequiresFullScreen` removed (deprecated); split view / Stage Manager supported. Hero/backdrop heights are clamped to a viewport fraction via `containerRelativeFrame(.vertical) { min(fixedHeight, length * 0.55…0.62) }` (iOS only — tvOS keeps fixed heights) in `HomeScreen.heroSection` / `LibraryHeroSection` / `MediaDetailScreen.backdropSection` so short windows keep content below the hero reachable. New full-bleed heroes must follow the same pattern.
- **Toolbar + Liquid Glass**: iOS 26 auto-renders `ToolbarItem` buttons with Liquid Glass. **Never** add `.buttonStyle(.glass)`/`.glassProminent` on toolbar items (nests double capsules). Active state via `.tint(themeManager.accent)` + `.fill` icon variant.

### Dependencies

`jellyfin-sdk-swift` v0.6.0, `Nuke`/`NukeUI` v12.9.0, `AVKit`/`AVPlayer`, `SwiftVLC` (libVLC 4.0, Swift 6 — [harflabs/SwiftVLC](https://github.com/harflabs/SwiftVLC), pinned **exactVersion 0.3.0**; ≥0.4.0 needs swift-tools 6.3 / Xcode 26.3+, toolchain is Swift 6.2.3 / Xcode 26.2; 0.3.0 is newest tag both 6.2-compatible AND shipping the PiP API — bump on Xcode 26.3+).

- `import SwiftVLC` works iOS + tvOS (`Player` is `@Observable @MainActor`; `VideoView`/iOS-only `PiPVideoView` are SwiftUI representables hosted in UIKit presenters via a child `UIHostingController`). Both `Cinemax` (iOS) and `CinemaxTV` (tvOS) targets link it — VLC is the default online playback engine on both.
- **RULE — Never re-add `VLCKitSPM`** — libVLC 3.x + 4.0 in one binary collide on `libvlc_*` C symbols. SwiftVLC replaced it (native PiP, object-based `Track` API fixing the audio-switch silence bug, Swift-6-native).
- **RULE — Adding a new file under `Shared/` requires re-running `xcodegen generate`** before it builds (the PostToolUse hook auto-regens only on `project.yml` edits, not new files; symptom of forgetting: "cannot find type X in scope" for a type that exists) Remote/Linux sessions (Claude Code on the web) have no xcodegen — CI still passes because it regenerates the project itself, but the committed `project.pbxproj` goes stale and local builds break; the CI "Verify committed project.pbxproj" step fails the PR until someone runs `xcodegen generate` on a Mac and commits the result.)
- **RULE — `@Observable` properties must NOT carry `didSet`/`willSet`**: the macro instruments setters via `withMutation { ... }`; a property observer body runs *after* the observation commits, and on collection-of-`Codable`-value-types properties (e.g. `[MenuEntry]`) the combo intermittently fails to deliver SwiftUI re-renders (symptom: bottom tab bar doesn't update after a reorder until the user taps anywhere). Pattern: keep stored properties plain `var T`, expose explicit `set*(_:)` mutator methods that mutate + persist, and use custom `Binding(get:set:)` in Pickers/Toggles so writes go through the mutator. Reference: `MenuConfigStore.setMode`/`setCustomKind` and the `Binding` projections in `MenuSettingsScreen+iOS.swift`.

### API protocol split (`Packages/CinemaxKit/.../APIClientProtocol.swift`)

`APIClientProtocol = ServerAPI & AuthAPI & LibraryAPI & PlaybackAPI & AdminAPI & SyncPlayAPI`. View models needing multiple domains take `APIClientProtocol`; leaf controllers narrow to a slice (`PlaybackReporter`/`SkipSegmentController` → `any PlaybackAPI`; `NowPlayingInfoController` → `any LibraryAPI`; `SyncPlayController` → `any SyncPlayAPI`). `AdminAPI` is a privilege boundary — gated on `AppState.isAdministrator`; server enforces authoritatively. `SyncPlayAPI` members all have default no-op implementations so test mocks compile without overriding.

### Swift 6 `nonisolated` escape hatches (safe when body only reads parameters)

1. `View, Equatable` sub-type inside a `@MainActor` screen needs `nonisolated static func ==` (`Equatable` isn't main-actor-isolated). See `PlayActionButtonsSection` in `MediaDetailScreen.swift`.
2. A `@MainActor` class's `static func` returning non-Sendable into a `TaskGroup @Sendable` closure needs `nonisolated private static func`. See `HomeViewModel.fetchGenreItems`.
3. A `nonisolated static func` reading a `static let` needs the constant `nonisolated` too (Sendable types only). See `SearchViewModel.sanitize` + `maxQueryLength`.
4. When `nonisolated static func ==` reads a non-Sendable DTO stored property, wrap the body in `MainActor.assumeIsolated { ... }` (safe — SwiftUI diffs on main actor). See the `MediaDetail*Section` / `MediaDetailEpisodeCard/Row` extractions.
5. **`@retroactive @unchecked Sendable` for SDK value enums**: the Jellyfin SDK doesn't annotate `BaseItemKind` (a `String`-raw enum). `@preconcurrency import` silences old warnings but Swift 6.1 region-based isolation still rejects `[BaseItemKind]?` crossing async/actor boundaries or being captured by `@Sendable` closures. Bridge in `Packages/CinemaxKit/.../Models/JellyfinSendable.swift`. Strictly scope these conformances to genuinely-safe value types (String-raw enum, no associated values).

## Design System (all RULE — `docs/design-system/conventions.md` rejection checklist is authoritative; read `docs/design-system/README.md` before editing UI)

- Tokens in `CinemaGlassTheme.swift`. All `CinemaColor` use `Color.dynamic(light:dark:)`. **Never `Color(hex:)` for new tokens.**
- **Shared toggle**: `CinemaToggleIndicator` (Capsule+Circle pill), parent-driven (`Button { value.toggle() }`). Never system `Toggle` in settings.
- **No 1px borders** — use color shifts. Glass panels: `.glassPanel()`.
- **Accent**: `themeManager.accent`/`.accentContainer`/`.accentDim`/`.onAccent` — never `CinemaColor.tertiary*`. All dual-mode.
- **Dark/Light**: always route through `themeManager.darkModeEnabled =` / `themeManager.accentColorKey =` — direct `@AppStorage` writes bypass `_accentRevision` and break reactivity. `.preferredColorScheme()` applied at root in `AppNavigation` only.
- **Hardcoded `.white`/`.black`**: only inside the video player (always dark) and on saturated `accentContainer`. Else `CinemaColor.onSurface`/`.onSurfaceVariant`.
- **Font scaling**: `CinemaScale.factor` = 1.4× base on tvOS × user `uiScale` (80–130%). **RULE — no bare `.font(.system(size: N))` numeric literals** — wrap `N` in `CinemaScale.pt(...)` or use a `CinemaFont.*` token, else it ignores `uiScale`/tvOS 1.4×. Exception: Play/Lecture labels hardcode 28pt on tvOS (computed var, not a bare literal).
- **tvOS focus**: `@FocusState` + `.focusEffectDisabled()` + `.hoverEffectDisabled()`, no white bg. Indicator = **3px accent `strokeBorder` + accent-tinted halo** (`cinemaFocus()`, was 2px@0.8 + a near-invisible grey `surfaceTint` glow). Cards **scale 1.06** + brightness on focus (`CinemaTVCardButtonStyle`); CTAs (`CinemaTVButtonStyle`) add an **accent focus ring + 3pt lift + ghost-fill brighten**; round portraits (`CastCircle`) draw their **own circular** accent ring (the rectangular `cinemaFocus()` is wrong for a circle). **RULE — the 1.06 card growth (~8pt on the 6-col/32pt-gutter tvOS grid, single-item focus) never overlaps neighbours, but every card-bearing scroll row MUST keep `.scrollClipDisabled()`** so the scaled edge card isn't clipped. Opt-in `dimUnfocusedPosters` (default off) dims non-focused posters for a spotlight effect — driven by a per-card `@FocusState` in `LibraryPosterCard`, no grid-level plumbing. Settings rows: `.tvSettingsFocusable()`. **Trait caveat**: a focused `Button` flips `UITraitCollection` to light-mode inside its label; `tvSettingsFocusable` takes `colorScheme` and injects on content + background shape — always pass `themeManager.darkModeEnabled ? .dark : .light`. **Hero `.focusSection()` rule**: any hero whose Play/More Info buttons sit in `.overlay(alignment: .bottomLeading)` of a tall `Color.clear` sizing block (Home + Library) needs `.focusSection()` on the buttons row AND the immediate row above (`tvTopBar` etc.) must also be a focus section — else up-presses get absorbed in the hero bounds.
- **iOS focus**: `.cinemaFocus()` (accent border + shadow).
- **CinemaButton styles**: `.accent` = primary CTAs (saturated `accentContainer` + `.white` text). `.primary` = neutral gradient (only on `DestructiveConfirmSheet`). `.ghost` = secondary (Retry, Clear Filters).
- **Motion Effects**: `motionEffectsEnabled` env key (from `@AppStorage("motionEffects")`). When off, all `.animation()` → nil.
- **Haptics SSOT** (`DesignSystem/Haptics.swift`, `@MainActor` enum): ALL app haptics route through it — single owner of `UIFeedbackGenerator`, collapses to no-op on tvOS. `tap` (light impact — toggles), `success`/`warning`/`error` (notification — toasts), `selection` (info toasts). Exception: the VLC player HUD owns its own feedback in the presenter files.
- **VoiceOver on custom toggles** (`SettingsRowHelpers.swift`): `CinemaToggleIndicator` is visual-only — every toggle row collapses to one element via `.accessibilityElement(children: .ignore)` + `.accessibilityLabel` + `.accessibilityValue(a11y.toggle.on/off)` + `.accessibilityAddTraits(.isToggle)` + an `.accessibilityAction` that toggles and fires `Haptics.tap()`. New custom toggles must follow the same pattern.
- Platform-adaptive via `#if os(tvOS)` or `horizontalSizeClass`.

## Navigation

- `AppNavigation` → Keychain session check → `apiClient.reconnect()` + `fetchServerInfo()`. Injects `ThemeManager`, `LocalizationManager`, `ToastCenter`, `MenuConfigStore`; applies `.preferredColorScheme()` at root.
- Flow: no server → `ServerSetupScreen` → `LoginScreen` → `MainTabView` (top tabs tvOS, sidebar iPad, bottom tabs iPhone).
- **RULE — All play buttons use `PlayLink<Label>`** (Button+coordinator on tvOS, `NavigationLink` on iOS) — never direct `NavigationLink` to `VideoPlayerView`.
- **Session expiry / 401 — confirm before logout (RULE)**: a single 401 must NEVER log the user out directly (cold-wake sockets, transient server hiccups, a flaky proxy all produce spurious 401s on a still-valid token — the Apple-TV "disconnected overnight" bug). `JellyfinAPIClient.setOnUnauthorized` → posts `.cinemaxSessionExpired`; `AppNavigation` now runs `appState.handlePossibleSessionExpiry()` which **silently re-validates** the token (`ServerAPI.validateSession()` → `GET /Users/Me`, returns `SessionValidity {valid|invalid|indeterminate}`) with a bounded 0/0.4/0.8s retry, **gated on `NetworkMonitor.isOnline`** (offline ⇒ never logout) and **debounced** (`sessionRevalidationInFlight`). Only a confirmed `.invalid` calls `logout()` + posts `.cinemaxSessionConfirmedInvalid` (the toast). `.valid` ⇒ `refreshCurrentUser()` and stay signed in; `.indeterminate` ⇒ keep session. Same coordinator drives proactive re-validation on `scenePhase .active` after a >60s background gap. Detection is the precise `JellyfinAPIClient.isUnauthorized(_:)` SSOT — pattern-matches `Get.APIError.unacceptableStatusCode(401)` (Get is a direct dep) + `JellyfinError.unauthorized` (raw PlaybackInfo POST) + `NSURLErrorUserAuthenticationRequired`; **never** the old `(401)` substring (false positives). 6 hot paths + `getPlaybackInfo` call `notifyIfUnauthorized`. **Keychain** items use `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` (readable on tvOS cold-boot relaunch; `migrateAccessibilityIfNeeded()` one-shot re-saves legacy items in `restoreSession`). **VLC player** observes background/foreground: a stream that died during sleep re-resolves a fresh `getPlaybackInfo(engine:.vlc)` and resumes from `positionAtBackgroundMs` (`reResolveAndResume`) instead of a terminal error.

### Custom menu / dynamic tabs (`MenuConfigStore`)

`MainTabView` consumes `menuConfig.resolvedTabs` (computed from `mode` / `customKind` / persisted entry arrays / `availableViews` cache). Three modes:

- **`.default`** — the canonical 5 tabs (Home, Movies, TV Shows, Search, Settings).
- **`.custom + .contentType`** — user picks which of the canonical 5 are enabled and in what order.
- **`.custom + .library`** — user surfaces individual Jellyfin libraries (returned by `getUserViews`) as tabs, filtered to video-bearing kinds (`movies`/`tvshows`/`homevideos`/nil **plus `boxsets`/`playlists`** — see `LibraryView.nonVideoCollectionTypes` blacklist; `music`/`photos`/`livetv`/`trailers`/`folders` excluded). `boxsets`/`playlists` are folders-of-folders, so they route to a different destination (see Collections/Playlists RULE below), not the flat-grid `MediaLibraryScreen`.

- **RULE — `MenuConfigStore.maxEnabledTabs = 5` caps the enable-*toggle*, NOT the resolved tab count**: the cap drives `applyCap` + the toggle refusal (`ToggleResult.refusedCapReached` → toast) so the *editor* won't push a menu past 5 enabled entries via the in-app toggles. But `.custom + .library` menus can still resolve to **>5 tabs** in practice (several libraries surfaced alongside Home/Search/Settings), and iPhone `TabView` in compact width then falls back to `UIMoreNavigationController` (the "More" overflow). **That overflow is tolerated, not a bug** (product decision) — a settled >5 library layout renders and navigates fine; its quirks are cosmetic: each More entry keeps its own nav stack (a pushed detail is retained when you revisit the entry) and a Back press can pop several levels at once. The hazard the cap actually guards against is a **live mutation crossing the 5↔6 boundary while the bar is on screen** (`UIScrollView does not support multiple observers … removing old observer <UIMoreNavigationController>`, user dumped back to the first tab) — a static overflow config never trips it. Don't "fix" a library menu into ≤5 tabs on the cap's account.
- **RULE — Library-mode tabs accept `parentId: String?` + `overrideTitle: String?` on `MediaLibraryScreen`** so each library tab scopes its queries to that view id. `BaseItemKind?` is nilable on `MediaLibraryScreen`/`MediaLibraryViewModel`: `nil` means "no `includeItemTypes` filter" — needed for `Mixed`/`Other` libraries where Jellyfin items aren't reliably typed.
- **RULE — Collections (BoxSets) + Playlists are folders-of-folders → `ResolvedTab.Destination.libraryFolders`**, not `.libraryView`. `MenuConfigStore.libraryTab` detects `collectionType == "boxsets"|"playlists"` and routes to `LibraryFolderBrowseScreen` (lists the folders; tapping one pushes a scoped `MediaLibraryScreen(parentId:)` for its contents). The flat-grid `MediaLibraryScreen` can't be used directly here — its cards open item *detail*, but a boxset/playlist card must drill into a *folder*. These appear only in `.custom + .library` menus on servers that expose Collections/Playlists views; not always-on tabs (keeps the 5-tab cap / focus machinery untouched).
- **RULE — Don't tag a tab with `role: .search`** on iPhone in this app: per Apple WWDC 2024, a search-role tab is force-placed at the trailing edge of the bar regardless of declaration order, which conflicts with user-reorderable menus and was the source of "redirected to Search" after a drag. Search is a regular `Tab` here.
- **RULE — Use `Tab(value:)` over `.tabItem + .tag` on both iOS and tvOS**: the pre-iOS-18 `.tabItem + .tag` pattern doesn't preserve child-view `@State` across a `ForEach` collection diff on tvOS — every `MenuConfigStore` mutation was remounting `SettingsScreen` and dropping its sub-nav before this migration. `Tab(value:)` builds identity off the `value` and survives collection mutations cleanly.
- **RULE — Settings sub-navigation state lives on `SettingsNavCoordinator`, not `SettingsScreen.@State`**: tvOS's `TabView` is bridged to `UITabBarController` which indexes child view controllers by **position**. Any layout shift in the tab bar (toggle off, reorder, kind change) — even with `Tab(value:)` — recreates the `UIHostingController` for the tab whose position changed, dumping the SwiftUI `@State` inside. `selectedCategory` / `selectedInterfaceSub` are stored on `SettingsNavCoordinator` (`@MainActor @Observable`) instantiated on `AppNavigation` (which never remounts during normal usage) and injected via `.environment`. `SettingsScreen` exposes thin computed-property pass-throughs (`get`/`nonmutating set`) so call sites keep using `selectedCategory = X`; the `$`-projection cases use `@Bindable var nav = settingsNav` + `$nav.selectedCategory`. iOS doesn't exhibit the position-remount bug but uses the same coordinator for symmetry.
- **RULE — `MainTabView` tvOS bar renders from a `@State displayedTabs` snapshot; iOS renders `menuConfig.resolvedTabs` LIVE (never freeze iOS)**: on tvOS, rendering off the live `@Observable` reconfigures the UIKit-backed bar on every fine-grained edit, which pulls focus off the row the user is touching onto the top-bar active pill (reads to the user as a "page reload"). The tvOS snapshot is **frozen while `settingsNav.selectedInterfaceSub == .menu`** so toggle / reorder / library-refresh edits keep focus stable inside the editor; **structural** mutations bypass the freeze and refresh live: `onChange(of: menuConfig.mode)` (covers Reset which forces `mode = .default`) and `onChange(of: menuConfig.customKind)` (entire entry set is being replaced). The bar catches up the moment the user backs out of the editor. iOS has no focus engine to disturb — freezing it too was itself a bug (edits looked like no-ops while the editor was open); the 5-tab cap keeps live iOS mutations inside `TabView`'s happy path.
- Persistence is explicit (`SettingsKey.menu*`) via mutator methods on the store, **never** through property `didSet` (see Architecture RULE on `@Observable`). `refreshAvailableViews()` is idempotent (`if views != availableViews { ... }`) to avoid spurious Observation cycles during user interaction. Mandatory IDs: `[settingsID]` only — Home and Search are non-mandatory but default-on.
- **MenuSettingsScreen tvOS Mode / Kind rows use a single `CinemaToggleIndicator`**, not two pills — same shape as `tvGlassToggle` in Apparence (Dark/Light Mode). Icon + label flip with state (`rectangle.stack.fill`/`Par défaut` ↔ `slider.horizontal.3`/`Personnalisé` ; `film.stack`/`Par type de contenu` ↔ `books.vertical.fill`/`Par bibliothèque`). Each toggle fires a `menu.refresh.confirm` toast on flip — the bar rebuilds (entries swap, focus jumps), and the pill is the only signal that explains the re-layout to the user. **Do NOT** toast on the "Refresh libraries" row: the row already conveys status via spinner + `lastFetchError` subtitle, a duplicate pill on every tap is noise.

## Server Setup & Login

Two-step pre-auth flow, shared mobile design (Server → Login feels like one journey). Discovery (`JellyfinServerDiscovery` + `ServerDiscoverySheet`): UDP broadcast port 7359; probes limited (`255.255.255.255`) **and** each interface's directed broadcast via `getifaddrs`. `scan()` auto-retries once after 800ms on empty (iOS local-network permission race). `NSLocalNetworkUsageDescription` in `iOS/Info.plist`. `AppState.disconnectServer()` → `ServerSetupScreen` ("Change server").

- **Quick Connect** (`QuickConnectSheet`, both platforms): passwordless sign-in — initiate → show 6-char code → poll until the user approves it from another signed-in session, then exchange for a session. `AuthAPI.isQuickConnectEnabled/initiateQuickConnect/quickConnectAuthorized/authenticateWithQuickConnect`; `QuickConnectRequest {code, secret}` model (secret never shown). `LoginViewModel.startQuickConnect` owns the poll loop (≈3s) + shared `completeSession`. The CTA on `LoginScreen` (both layouts) is **gated on `isQuickConnectEnabled()`** (hidden when the server has it off / pre-10.x). iOS `.sheet`, tvOS `.fullScreenCover`. Biggest win on tvOS — no password typing on a remote.
- **Quick Connect authorize side** (`QuickConnectAuthorizeSheet`, Settings → Account): a signed-in user types the 6-char code another device shows on its login screen; approving completes that device's sign-in. iOS `.sheet` / tvOS `.fullScreenCover` (tvOS `.sheet` renders cramped). iOS auto-focuses the code field; tvOS deliberately does NOT (would force-open the full-screen keyboard over the instructions).
- **`ServerHelpSheet`** ("How do I find my server?", from `ServerSetupScreen`): static help content. **RULE — chrome is platform-branched**: iOS `NavigationStack`+toolbar; tvOS full-screen cover with custom header + accent close button, because tvOS `.toolbar` on a modal clips the title and renders the trailing button as an empty pill. `PrivacySecurityScreen` uses the same pattern (iOS `.sheet`, tvOS `.fullScreenCover`); when the modal's scroll content has focusable rows, the tvOS header needs `.focusSection()` or up-presses never reach the Done button.

- **RULE — LoginScreen mobile caveat**: ServerSetupScreen's `.padding(.horizontal, spacing4)` outside `.glassPanel` is silently dropped in `LoginScreen.mobileLayout` under iOS 26. Workaround: `.frame(maxWidth: formMaxWidth)` (350pt) on form panel + actions VStack. Don't "fix" without pixel-sampling.
- **Rainbow easter egg**: top icon block is a `Button` → `AccentEasterEgg.tap(…)` (resolver in `SettingsScreen.swift`). Cycles `AccentOption.cyclingCases`; unlock flips `@AppStorage(SettingsKey.rainbowUnlocked)`. `ThemeManager` checks `isRainbow` first → HSB driven by `_rainbowHue`.

## Media Library (`MediaLibraryScreen`)

Unified, parameterized by `BaseItemKind`. State: `LibrarySortFilterState` (default `dateCreated` desc).

- **Browse vs filtered**: browse layout (hero + genre rows + browse-genres grid) when `!isFiltered`; any filter → flat grid. `isFiltered` = genre chips OR `showUnwatchedOnly` OR `selectedDecades`. Both platforms honor `library.tvBrowseLayout` (`grid` forces flat grid even unfiltered) — the key keeps its legacy `tv` name but is platform-neutral (`LibraryBrowseLayout` enum; preserves existing on-device prefs).
- Title count uses `isFiltered` (not `isNonDefault`). `loadInitial` guarded by `hasLoaded` (prevents re-randomization on tab switch); `reload(using:)` bypasses (pull-to-refresh, `.cinemaxShouldRefreshCatalogue`).
- **RULE — `performLoad` renders progressively**: `isLoading` flips false right after the phase-1 genres + hero fetch (NOT after `fetchGenreItems`), so the hero paints while the genre rows fill in off `itemsByGenre`. ## Video Playback

### Playback engine (VLC default — fixes the MKV/Dolby-Vision freeze)

Online playback defaults to VLC on iOS + tvOS. Settings → Interface **"Use Native Player"** (`SettingsKey.forceNativeAVPlayer`, default `false`) falls back to `AVPlayer`/`NativeVideoPresenter`.

**Why:** `AVPlayer` can't open MKV → Jellyfin forced into a 4K HEVC + DV→SDR re-encode the server can't do real-time → frozen playback. No `AVPlayer` device profile makes Jellyfin remux DV-in-MKV (DV only passes through on DirectPlay/DirectStream, never in a transcode). VLC DirectPlays the raw file → zero server transcode, 4K/HEVC/DV preserved.

- **RULE — Device profile split** (`JellyfinAPIClient+Playback.swift`): `getPlaybackInfo(... engine:)`. `.vlc` → `buildVLCDeviceProfile` (one broad DirectPlayProfile, **no container restriction** → `/Videos/{id}/stream?static=true`, no transcode). `.native` → `buildAppleDeviceProfile` (AVKit-safe codec profiles). **API default is `.native`** so internal AVPlayer-only calls are unaffected; engine chosen explicitly at `VideoPlayerCoordinator` (tvOS) / `VideoPlayerView` (iOS) from `forceNativeAVPlayer`.
- **RULE — seek-heavy containers force a server HLS transcode on the VLC path** (`isSeekHeavyContainer`: `avi/divx/wmv/asf/flv/vob/mpg/mpeg/mpe/m2v`): these keep their index at EOF, storming libVLC/origin with byte-range requests (raw AVI/XviD froze on the reverse-proxied server on BOTH direct and loopback-proxy paths — true remux impossible for XviD). On failure the code falls back to the raw stream. **The transcode profile deliberately includes a broad audio-codec allow-list** so the server does NOT re-encode audio to AAC — omitting it caused a constant audio offset ("son décalé") on transcoded AVI/XviD.
- **`VLCStreamPresenter`** (iOS+tvOS): UIKit `UIViewController` shell + all controllers/HUD; SwiftVLC `events` `AsyncStream<PlayerEvent>` consumed in one `@MainActor` Task. Rendering: child `UIHostingController` hosting the shared `PlayerEngineSurface` — `PiPVideoView` on iOS (native PiP for ALL content incl. MKV/DV via `pip.enter` → `PiPController.toggle()`), `VideoView` on tvOS (no PiP — tvOS has none). Time strings via shared `PlayerTimeFormat`; leaf transport views live in `PlayerTransportViews.swift`. Object-based track selection (no silence bug); Jellyfin defaults applied once on `.tracksChanged` by ordinal-within-type. **libVLC 4.0 has no distinct `.ended`** — end-of-media is `.stopped`, disambiguated via `isTearingDown` flag + `lastPlayStart` (>1s) + near-end guard. Auth via `api_key` **query param** (libVLC can't reliably inject the header). Full feature parity. **AirPlay-to-TV video** still impossible on any libVLC path — deferred.
- **RULE — broken-IPv6 servers route through a loopback proxy** (`CinemaxStreamProxy.swift`): libVLC's HTTPS access (`vlc_tls_SocketOpenTLS`) hardcodes `AF_UNSPEC`, ignores the `ipv4`/`ipv6` options, and has **no Happy-Eyeballs** — so a dual-stack server whose IPv6 (AAAA) is black-holed makes libVLC stall ~75s on the dead route (`URLSession` is immune; it races families). No libVLC media flag can force IPv4. `StreamTransportPolicy.shared` (`@MainActor` singleton) probes once in the background (from `AppNavigation` on launch / server change / foreground / `network.isOnline` flip): resolves the host's AAAA, opens a short NWConnection (TLS, SNI=host) to that IPv6, and sets `preferProxy` only if it **hangs** (neither `.ready` nor fast `.failed` within 4s — the genuine black-hole; an IPv4-only network fast-fails and must NOT proxy). When `preferProxy`, `startPlayback` opens `http://127.0.0.1:<port>/…` and `CinemaxStreamProxy` re-fetches each request from the real HTTPS origin via `URLSession` (Happy-Eyeballs → IPv4), streaming back with `Range` preserved. A watchdog→proxy fallback also catches black-holes the probe missed (the retry stays on the proxy, never the broken direct URL). Healthy servers keep the **byte-identical direct path** — the proxy only runs when `preferProxy` or a first direct attempt already failed. **RULE — the loopback handler (`forward`) rejects any request that isn't a well-formed `/s/<id>` path or whose `Host` header (when present) isn't loopback** (400) — defense-in-depth on top of the 127.0.0.1-only bind; every legitimate stream URL is `http://127.0.0.1:<port>/s/<uuid>` so this never touches the real path.
  - **RULE — the proxy's `URLSession` MUST use a CONCURRENT delegate queue** (`maxConcurrentOperationCount=8`, not the default serial one). The proxy applies backpressure by **blocking** `didReceive(data)` until each loopback `NWConnection.send` lands; MKV demuxing fires simultaneous `Range` requests (reads the SeekHead/Cues near EOF *while* the `bytes=0-` stream is still open). On a serial queue the main stream's blocked callback head-of-line-blocks the seek request → VLC times out the seek → `libvlc stream filter error: cannot seek` + `damaged file?`. URLSession still serializes callbacks per-task, so a concurrent queue is safe. One request per connection (`Connection: close`); each loopback URL carries a unique `/s/<id>` resolving to its own target so a retry/episode swap can't read the wrong stream.
  - **RULE — transparent mid-stream reconnect** (`UpstreamHandler`): a reverse-proxied / HTTP-2 origin routinely RSTs a long-lived range request mid-stream. On a non-cancelled error AFTER the head was sent, the handler re-issues the origin GET at `rangeStart + bytesDelivered` (a `Range` request) and **splices the body into the SAME loopback connection** — libVLC never sees EOF, just a brief feed pause absorbed by `:network-caching`, so a transient drop is **invisible** (this is the controllable version of `:http-reconnect`, which was removed because it hung libVLC silently). The reconnect's response head is NOT forwarded; a non-206 (origin ignored `Range`) bails to a clean close so libVLC re-opens. Bounded by `reconnectsLeft`, which renews ONLY after `progressRenewBytes` (256 KB) of progress since the last reconnect — a flaky-but-feeding stream recovers forever, a trickle-then-RST loop depletes the budget and gives up. `task` is lock-guarded (read on netQueue, written on the delegate queue during a reconnect); the other counters stay lock-free (per-task callbacks are serialized and strictly sequential across a reconnect). `finish()` is one-shot (a `.cancel` disposition re-fires `didCompleteWithError`).
- `PlaybackReporter` engine-agnostic via optional `TimeSource` closure (AVPlayer → nil reads `Context.player`; VLC injects VLC time).
- **VLC player extras (both platforms unless noted)**: trickplay scrub previews (`TrickplayController` — manifest rides the same `getItem` the chapter strip does; tiles auth like chapter thumbs; silently absent when the server hasn't run the Trickplay task); speed picker 0.5×–2× (`playbackRate` reapplied on `.playing` — libVLC resets rate per media; iOS long-press on video = hold-to-2×); audio/subtitle delay nudge pickers (`player.audioDelay/subtitleDelay`, µs `Duration`s, reset per media); stats overlay (1s-tick repaint; libVLC bitrates shown ×8000 as kb/s, matching VLC's own UI); next-episode countdown card during the outro segment (replaces Skip Credits when autoplay armed; Cancel sets `nextUpCancelledForThisItem`, also checked by `handlePlaybackEnded`). **tvOS info panel**: swipe-down (indirect) while HUD hidden; Menu press order is panel → HUD → dismiss; `presentPicker`/`endPicker` skip `showControls()` while the panel is up so the two chrome layers never stack. **Loading spinner** (`loadingIndicator`, centered `UIActivityIndicatorView`) shows while the engine is opening / (re)buffering / on a retry (driven by `.opening`/`.buffering` state + each (re)open call site) and hides the moment frames flow — a stall or reload reads as "loading", not a frozen frame.
- **tvOS transport is custom** (no `AVPlayerViewController`): `TVScrubBar` (focusable; clickpad left/right = ±10s only while focused), focusable control bar (Prev/Next/Audio/Subtitles — **no on-screen play/pause**, center glyph flash feedback), always-on chapter strip (peeks 40pt / expands 178pt on focus). `ChapterChip` draws own focus state. **Menu press peels one layer at a time** (matches system `AVPlayerViewController`): info panel open → close it; HUD visible → hide it (and cancel the auto-hide timer); bare video → `dismiss`. **RULE — Menu peel handled by `handleMenu()`, called from BOTH a `UITapGestureRecognizer(allowedPressTypes:[.menu])` on `view` AND `pressesBegan`, keyed on `controlsContainer.alpha` (the VISUAL truth, never the `controlsVisible` flag)**: tvOS routes the Menu press to different responders by focus state — while the HUD is visible `tvScrub` holds focus (`preferredFocusEnvironments`) so the recognizer wins over the system's default modal-dismiss; while the HUD is **hidden** no control is focused and `pressesBegan` is the only path that fires (same reason `handleRemotePlayPause` exists). Handling it in only one place left a gap: a HUD-hidden press fell through `pressesBegan`'s "reveal the HUD" branch → Menu re-opened the HUD instead of quitting (an infinite hide/reveal loop). `handleMenu` is called from both paths and debounced via `lastMenuHandledAt` (a single press can reach both within one event) so it acts exactly once. Keying on `controlsVisible` (a flag that can disagree with the on-screen alpha) was the original loop's root cause — always read `controlsContainer.alpha`. **RULE — the "wake a hidden HUD" branch is a whitelist (`hiddenHUDIntent(for:)`), NOT "any press"**: the tvOS *simulator* delivers a keyboard Escape as a spurious non-`.menu` press (observed `type.rawValue` 2041) that, under "any press reveals", un-hid the HUD one event *before* the real Menu press (type 5) arrived — so the Menu peel saw `alpha==1` and hid instead of dismissing, never quitting. `pressesBegan` treats `key?.keyCode == .keyboardEscape` as Menu (handled FIRST, early-return). Real remotes send a clean type-5 Back press, but the whitelist keeps the simulator (and any future stray press type) from breaking the peel — an unrecognized press maps to `nil` and is ignored. **A recognized wake press ALSO performs its action, not just reveal**: select/Enter → play/pause, left/right → skip ∓N s, up/down → reveal only. **`hiddenHUDIntent` matches BOTH Siri-remote `UIPress.PressType` AND hardware-keyboard `press.key?.keyCode`** — keyboard arrows/Enter arrive only as `key.keyCode` (no arrow `PressType`), so before this they did nothing on a hidden HUD; `revealControls()` then moves focus to `tvScrub` so the next press routes normally. After every programmatic seek `refreshTimeUISoon()` repaints (VLC emits no time updates while paused; it cancels its prior follow-ups so a scrub burst can't stack closures).
- **RULE — tvOS touch-surface scrubbing** (`TVScrubBar`): Siri Remote touchpad slide = variable scrub via an **indirect-only** `UIPanGestureRecognizer`. Must be **incremental** — consume `translation(in:)` then `setTranslation(.zero)` each `.changed` and accumulate; an absolute baseline gets stomped by the periodic tick → "jump then snap" glitch. Sensitivity is the `scrubGain` constant (0.2 = full swipe ≈ a fifth of the timeline; tune here — was 0.5, too fast on the Siri Remote 2nd-gen touch surface). A scrub only engages after the finger crosses the `activationThreshold` dead-zone (30pt) — the touch surface fires a pan on the faintest contact, so without it a resting thumb both started scrubbing and swallowed the ±N clickpad presses (`pressesBegan` bails while `isScrubbing`); a sub-threshold touch is a no-op. No engine seek during the drag (preview labels only); one `engineSeek` on release. `VLCStreamViewController.pendingScrubTargetMs` (shared by scrub release AND coalesced ±N skips / chapter jumps — see the seek-coalescing RULE) holds the bar/labels at the target until VLC's async seek actually lands (±1.2s) so the tick can't paint the stale pre-seek position. `refreshTimeUI` honors it on BOTH platforms now (via the shared `paintPosition`), not just tvOS. Presses ignored while `isScrubbing`; `refreshTimeUI` returns early while scrubbing on both platforms.
- **RULE — ±N skips are coalesced into one debounced seek** (`seek(bySeconds:)` → `accumulateSeek` → `commitPendingSeek`): EVERY skip entry point (iOS skip buttons, iOS double-tap, tvOS clickpad ±10s, chapter-chip taps) accumulates an ABSOLUTE target into `pendingScrubTargetMs` and commits ONE `engineSeek` `seekCommitDelay` (0.3s) after the LAST press — never an immediate `player.seek(by:)` per press. Mashing the button used to fire N relative seeks, each tearing down + reopening libVLC's HTTP byte-range request → a storm that overloads a self-hosted / reverse-proxied origin and can stall the stream. **Don't "fix" the 0.3s delay away** — the HUD jumps to the projected position instantly (`paintPosition`) so it feels responsive; the delay is what collapses N requests into 1. The pending seek is cancelled (`cancelPendingSeekCommit`) on scrub takeover, media reload, episode swap, and teardown so a stale target can't seek the wrong position / a fresh episode. The pure seek math lives in `SeekCoalescer` (`clamp(target:lengthMs:)` / `relativeTarget(...)`, `endGuardMs = 250` — never seek within 250ms of length, libVLC treats end-seek as EOF), extracted for unit testing (`SeekCoalescerTests`); the view controller keeps debounce/HUD/engine ownership.

Independent of the gesture/HUD stack.

### SyncPlay / Watch Together

`SyncPlayController.shared` (`@MainActor @Observable` singleton, `Shared/Screens/VideoPlayer/SyncPlayController.swift`) owns the whole session: current group, participants, the realtime socket, the client↔server `clockOffset`, and a `PlaybackBridge` injected by the player surface. Depends on `any SyncPlayAPI` + `LocalizationManager` + `ToastCenter`, set on activation.

> **STATUS — UI hidden (as of 1.0.5)**: Watch Together is not production-ready, so **both UI entry points are gated off** behind `MediaDetailScreen.watchTogetherEnabled` — a `private static let = false` compile-time kill-switch. The button doesn't render on either platform and no group can form, so everything documented below (controller, socket, `WatchTogetherSheet`, VLC HUD participant pill) still compiles and is type-checked but is **unreachable at runtime**. Re-enable by flipping that flag to `true`; revisit after 1.0.5 ships.

- **API**: the Jellyfin SDK doesn't model SyncPlay — requests are hand-built `URLSession` (mirroring `rawPostPlaybackInfo`): `MediaBrowser` auth header, `setEndpointPath` (base-path RULE), 15s timeout, `notifyIfUnauthorized`. REST: `GET /SyncPlay/List`, `POST /SyncPlay/New|Join|Leave|Pause|Unpause|Stop|Seek|Ready|Buffering|SetNewQueue`, `GET /GetUtcTime`. Realtime: `SyncPlaySocket` (a Swift `actor` around `URLSessionWebSocketTask`, `wss?://…/socket?api_key=…&deviceId=…`, parsed `Sendable` frames via `AsyncStream`).
- **Group lifecycle**: `WatchTogetherSheet` (`MediaDetailWatchTogether.swift`) → `createGroup`/`joinGroup`. The socket comes up FIRST so the client hears the server's authoritative `GroupJoined` echo; create sets an optimistic placeholder refined by that echo. **v1 ties group lifetime to the player** — `playbackDidDismiss()` leaves the group on teardown.
- **Sync enforcement**: local taps never move the playhead directly. In a group, user play/pause/seek go to the server (REST); the server's `SyncPlayCommand` echo over the socket is what actually moves the playhead — all participants stay in lockstep. Commands carry a server-UTC `When` converted via NTP-style clock offset (3× `GetUtcTime` averaged, ~100ms accuracy); far-future timestamps capped at 30s. `Ready`/`Buffering` reports feed drift correction. **Transport-only** — content selection is NOT synced; each participant opens the item themselves.
- **RULE — bridge closures call the engine directly and must NEVER re-emit** (an inbound command must not loop back out); `isApplyingRemoteCommand` suppresses buffering/ready echoes while a remote command is applied.
- **RULE — `startSocket` snapshots the previous socket before the async stop** — racing the reassignment kills the NEW socket. `SyncPlayDateParser` uses `Date.ISO8601FormatStyle` (Sendable), never `ISO8601DateFormatter` (race across actor/URLSession paths).
- **Platform reach**: the sync binding exists ONLY in `VLCStreamPresenter` — `NativeVideoPresenter` has NO SyncPlay integration, so sync works only when playback routes through libVLC. UI entry (**currently gated off** — see STATUS above): `MediaDetailScreen` `person.2.fill` chip (iOS secondary actions) / tvOS action row, online-only, accent when `isInGroup`; VLC HUD shows a participant-count pill.

### Flow (native/AVPlayer path)

1. `getItem()`. 2. Resolve non-playable: Series → `getNextUp()` or first episode; Season → first episode (**must resolve to Episode** — Series/Season have no media sources). 3. - **Widget kinds** (all in `Widgets/CinemaxWidget/CinemaxWidget.swift`, families `.systemMedium`/`.systemLarge`, 4-column poster grid with a trailing "See all" tile): **Continue Watching** (`/UserItems/Resume`), **Favorites** (`/Items?isFavorite`), **Next Up** (`/Shows/NextUp`), **Recently Added** (`/Items/Latest` — bare array, not the `{Items:[]}` envelope, so its own decode). All share one `CinemaxRailKind` enum + `PosterRailProvider` + `PosterRailWidgetView` pipeline; adding a kind = one enum case + one `loadEntry` dispatch + one `JellyfinLite.fetch*` + one `Widget` struct registered in `CinemaxWidgetBundle`. **RULE — widget display strings are inline `french ? … : …`** (the extension can't link the app's `LocalizationManager`), keyed off `Locale.preferredLanguages.first?.hasPrefix("fr")`.
- **RULE — every binary needs its own `PrivacyInfo.xcprivacy`** (Apple's ITMS-91053 scan is per-`.app`/`.appex`): the app manifest is `Resources/PrivacyInfo.xcprivacy`; each extension carries its own (`Widgets/CinemaxWidget/PrivacyInfo.xcprivacy`, `TopShelf/CinemaxTopShelf/PrivacyInfo.xcprivacy`) — auto-bundled because each extension dir is already a `sources:` path. UserDefaults reason codes: the extensions declare `1C8F.1` (they touch ONLY the App-Group suite via `UserDefaults(suiteName:)` — `CA92.1` explicitly excludes data shared across binaries); the app declares BOTH `CA92.1` (its own `@AppStorage`/`UserDefaults.standard`) and `1C8F.1` (App-Group publish in `ExtensionSessionBridge`); the app manifest ALSO declares `NSPrivacyAccessedAPICategoryFileTimestamp` reason `C617.1` for statically-linked Nuke's disk cache (`.contentAccessDateKey`, ships no manifest of its own). A new required-reason API on any target = update that binary's manifest. Validate edits with `plutil -lint`.
- **RULE — extensions do NOT link CinemaxKit** (widget memory budget; SDK pulls Nuke + generated entities). They read the session the app publishes via `ExtensionSessionBridge` (CinemaxKit/Persistence) and talk to Jellyfin endpoints directly (`JellyfinLite` in the widget, inline in `ContentProvider`). Publish hook: `AppState.refreshCurrentUser()` (covers login / switch / restore) + clear in `logout()`.
- **RULE — extension session store is a shared Keychain group, NOT plaintext UserDefaults**: the token lives in a **shared Keychain access group** `$(AppIdentifierPrefix)com.cinemax.shared` (`kSecAttrService = com.cinemax.jellyfin`, `kSecAttrAccount = extension_session`, accessible `AfterFirstUnlockThisDeviceOnly`) — device-only, never in backups. This replaced the old plaintext App Group `UserDefaults` copy (suite `group.com.cinemax.shared`, key `extension.session`), which the token was readable-at-rest from (backups / any group process). `ExtensionSessionBridge.publish` **dual-writes both** as a one-release transition; **the UserDefaults write is a fallback to DROP next release** (TODO in-code) — never let the token sit in UserDefaults long-term. Readers (`KeychainService.readSharedSession`; inline in `JellyfinLite`/`ContentProvider`) try Keychain first, UserDefaults second. The full group string is resolved at runtime from the `AppIdentifierPrefix` Info.plist key (injected via `$(AppIdentifierPrefix)` on all 4 targets — avoids hardcoding the team id). **Keep BOTH contracts in sync across all three copies**: the Keychain service/account/group AND the UserDefaults suite/key/JSON shape (the extensions re-declare the literals; only the app links `KeychainService`). `keychain-access-groups` entitlement is on all 4 targets (xcodegen-generated); existing app-private Keychain items (`server_url`/`user_session`/`access_token`/`device_id`) are untouched so login is unaffected. **Release: enable "Keychain Sharing" on the 4 App IDs before an App Store build** (transparent with local automatic signing).
- **Deep links**: `cinemax://item/{id}` (scheme registered in both Info.plists; widget posters + tvOS Top Shelf both emit it). `AppNavigation.onOpenURL` → `AppState.handleDeepLink` → `pendingDeepLinkItemId`. **RULE — deep-link routing is tab-config-independent**: `MainTabView` switches to the Home tab when one exists (`HomeScreen` consumes it via a screen-root `navigationDestination(item:)` — lazy-container RULE applies); when the custom menu has **no** Home tab, `MainTabView` instead presents `MediaDetailScreen` modally from the tab root (`deepLinkFallback` token → iOS `.sheet` / tvOS `.fullScreenCover`, its own `NavigationStack`, tvOS re-injects `playerCoordinator` + `.onExitCommand` dismiss) so the link is never silently dropped. The `cinemax://home` "See all" tile still just switches tabs. **RULE — the item id is shape-validated before dispatch** (`AppState.isValidItemId`): accepts ONLY the two Jellyfin forms — a 32-char undashed hex string OR a canonical dashed GUID — and drops anything else, so a malformed deep link can't drive a lookup with attacker-controlled path text.
- **RULE — `TVTopShelfContentProvider` under Swift 6 strict concurrency**: use the completion-based `loadTopShelfContent(completionHandler:)`, wrap the handler in an `@unchecked Sendable` final-class box, and keep the `Task` closure body minimal (one `await Static.run(handler:)` call). Returning the non-Sendable `TVTopShelfContent` from the async override — or inlining the guard/await flow in the Task closure — trips a region-isolation checker bug ("Please file a bug", Swift 6.2.3).
- **RULE — TopShelf target MUST link `OTHER_LDFLAGS: -e _NSExtensionMain`**: the tv-app-extension product type's default entry `_TVExtensionMain` is an **empty `ret` stub in the tvOS 26 simulator runtime** — main returns instantly and dyld exits the extension (code 3) before the provider is instantiated, with zero logs and no crash report; HeadBoard silently falls back to the static Top Shelf image (verified in lldb: `TVServices\`TVExtensionMain: ret`). Foundation's generic `NSExtensionMain` drives the NSExtension lifecycle correctly. Diagnosis kit if it regresses: `boot.c` C-constructor log + `init`/invoke logs (subsystem `com.cinemax`, category `TopShelf`), trigger HeadBoard on demand with `xcrun simctl spawn <udid> launchctl kickstart -k system/com.apple.PineBoard`.
- **RULE — WidgetKit `TimelineProvider` witnesses use the same HandlerBox pattern** (NOT `@escaping @Sendable` parameter annotations): the SDK's completion annotations differ between toolchains — `@Sendable`-annotated witnesses built on Xcode 26.2 fail CI's Xcode 26.5 with the `sending`-closure error. The `@unchecked Sendable` box + minimal `Task { await Self.deliver…(handler:) }` body compiles on both.

## Settings Screen

Three-level navigation. Landing — tvOS: split (left brand, right nav pills, persistent accent bloom); iOS: vertical scroll + `NavigationStack`. Detail pages (Appearance/Account/Server/Interface) — tvOS `ScrollView` + back button (`onExitCommand`), iOS pushed. **Interface is itself a hub** of sub-pages (Main Menu / Home page / Detail page / Playback / Debug — `InterfaceSubcategory` enum); iOS routes via a second `.navigationDestination(item: $selectedInterfaceSub)` on the landing scroll; tvOS uses the same state-machine pattern as the category level. Interface hub uses the same pill chrome as the Settings landing (`iOSInterfaceSubButton` mirrors `iOSCategoryButton`, with `accentContainer` fill on the first row).

- **RULE — `MenuSettingsScreen+iOS` uses native `List` + `Picker(.segmented)` + `Stepper` + forced `editMode .active`**: these are deliberate exceptions to the "never system `Toggle` in settings" rule. The user explicitly requested native iOS chrome for the Mode/Kind selectors (Picker), font-size (Stepper), and always-visible drag handles (forced edit mode → native ≡ grip with no `EditButton` needed). Per-row enable/disable still uses `CinemaToggleIndicator` Button with `.buttonStyle(.borderless)` so the List cell doesn't intercept taps. Add new system controls in Settings only with the same explicit user mandate.
- **RULE — Settings detail screens (`SettingsScreen.iOSLayout`) must NOT wrap their body in a nested `NavigationStack`**: `MainTabView`'s `Tab` block + the `MoreTabScreen`-style overflow path each already provide one, and a third nested stack silently breaks `.navigationDestination(item:)` routing for sub-page pushes (Interface → Main Menu was unreachable through the More overflow until this was fixed). Pushed destinations attach to whatever stack is closest above.
- **RULE — `SettingsScreen.tvOSLayout` lifecycle gotchas**: SwiftUI re-presents this view whenever the `TabView` reorders (every `MenuConfigStore` mutation), even with the `SettingsNavCoordinator` keeping sub-nav alive. Two consequences must be handled inside this surface:
  - `.task { loadServerUsers() }` must guard against re-fires with `@State serverUsersLoadAttempted` — otherwise every menu edit re-hits `getUsers`/`getPublicUsers` and the failure path surfaces a toast on every interaction. Failure also silently falls through (no toast); `UserSwitchSheet` does its own fetch when the user actually opens it.
  - **Never** `.onAppear { proxy.scrollTo("settings.top") }` on the outer `ScrollViewReader` — it fires on every re-presentation and yanks the page to the top, dropping tvOS focus to the active tab pill. The two `.onChange(of: selectedCategory/Sub)` handlers already cover the only legitimate "scroll back to top" case (popping out of a category / sub-page detail).

- **RULE — tvOS focus**: each row is a single focusable unit (never individual sub-items). Accent/Language rows: left/right or select cycles (`onMoveCommand`).
- **Settings row SSOT** (`Settings/SettingsRowHelpers.swift` + platform extensions): every boolean toggle declared once as `SettingsToggleRow`, rendered both platforms from same list. Catalogues: `interfaceToggleRows`/`homePageToggleRows`/`detailPageToggleRows`/`debugToggleRows`. Adding/renaming a toggle = one-line edit. Expanders: `iOSToggleRowsJoined(_:accent:animated:)`, `tvToggleList(_:)` (ignores `row.tint`; `tint:` is iOS-only debug-orange). `tvActionRow(...)`, iOS atoms (`iOSSettingsRow` etc.), tvOS `tvGlassToggle`.
- **Assets**: `AppLogo.imageset` — iOS `app_logo.png` (full); tvOS `app_logo_tv.png` (front parallax layer, transparent bg, no `clipShape`).
- **Watch History** (`WatchedHistoryScreen`, Account section row → `showWatchedHistory`): played movies + episodes, `getItems(filters:[.isPlayed], sortBy:[.datePlayed] desc, includeItemTypes:[.movie,.episode])`, paginated 40/page via `PaginatedLoader` (mirrors the library filtered grid — `.onAppear` last-card trigger, footer spinner). Grid + chrome mirror `FavoritesScreen`/`PrivacySecurityScreen`: iOS `.sheet` (`NavigationStack` + toolbar Done); tvOS `.fullScreenCover` (custom header + accent Done + `.focusSection()`; the always-present Done button is the focusable that loading/empty/error states need). Cards carry `.mediaCardContextMenu` (un-watch drops the item on reload). Keys `settings.watchedHistory`, `watchedHistory.empty.{title,subtitle}`.
- **Quick user switch**: `UserSwitchSheet` — grid → password → re-auth, `apiClient.reconnect(url:accessToken:)` without clearing server URL. **RULE — user list has three fetch outcomes**: `getUsers()` is admin-only (401 for regular accounts), `getPublicUsers()` returns `[]` on hardened servers where the welcome screen is disabled (Jellyfin's secure default). When both yield nothing, the sheet falls through to an informative empty state (`switchAccount.noPublicUsers.*`) with a "Connexion manuelle" CTA → `manualEntryStep` (username `TextField` + password `SecureField` → same `authenticate(username:password:)` path). Also surfaced as "Utiliser un autre compte" under a populated grid for hidden-account login. Removing the manual path silently regresses the App Store reviewer scenario.
- **Refresh Catalogue (single trigger)**: Settings → Server `apiClient.clearCache()` + posts `.cinemaxShouldRefreshCatalogue` (Home + MediaLibrary observe). **No per-page refresh buttons** — Settings is SSOT. iOS also `.refreshable`.
- **RULE — two-tier catalogue refresh** (`Notification.Name` in `AppNavigation.swift`): **tier-1 `.cinemaxShouldRefreshCatalogue`** = catalogue *content* changed / cache cleared → FULL reload (Home `reload()`, MediaLibrary genre fan-out). Posted by Settings "Refresh Catalogue", parental-controls rating limit (`PrivacySecurityScreen.ageRow`), and all Admin metadata / identify / delete flows. **tier-2 `.cinemaxItemUserDataChanged`** = ONE item's watched / resume-position userData changed via a per-item toggle → *targeted* refresh. Posted by `MediaCardContextMenu` (watched), `MediaDetailViewModel` (item/season/episode watched toggles), `HomeViewModel.mutateResumeItem` (Continue Watching), and `PrivacySecurityScreen.clearContinueWatching`. Favorite hearts stay on their own `.cinemaxFavoritesChanged` fast path (untouched). **Consumers**: Home runs `HomeViewModel.refreshUserDataRails()` on tier-2 (resume + next-up + favorites only — NO genre fan-out / latest / sessions, and like `refreshResume` it does NOT rebuild the episode-nav maps); MediaLibrary reloads for BOTH tiers (a grid context-menu toggle must reflect under the unwatched-only filter); Favorites + WatchedHistory reload for both. **Deferral**: Home + MediaLibrary track `@State isVisible` (flipped in `.onAppear`/`.onDisappear`) — while hidden inside `TabView`, a notification only sets a pending flag and the reload runs on next `.onAppear` (a pending full reload subsumes a pending targeted one; several tier-2 events coalesce into one). Degrades safely: if `onDisappear` never fires, `isVisible` stays true and behavior is the old immediate reload. `mutateResumeItem` no longer calls `refreshResume` directly — it posts tier-2 and Home's own handler fetches the resume rail exactly once (no double-fetch).
- **Connected devices** (`PrivacySecurityConnectedDevicesList`, from Privacy & Security): lists every signed-in device (`getDevices`) with per-row destructive sign-out (`deleteDevice` + confirmation). **RULE — the current device** (matched via `KeychainService.getOrCreateDeviceID()`) is highlighted separately with the revoke action hidden — revoking it would log this device out. **RULE — a tvOS screen pushed inside a `.fullScreenCover` must have ≥1 focusable view in EVERY state** (loading/empty/inert cards get `.focusable()` + `.focusEffectDisabled()`): with zero focusables, focus stays on the covered root screen, so a Menu press hits the root's `.onExitCommand` and tears down the whole cover instead of popping the push.
- **Debug section** always visible (not `#if DEBUG`-gated) so QA doesn't need a custom build. Icons orange.

## Network awareness

- **`NetworkMonitor`** (`@MainActor @Observable` around `NWPathMonitor`): seeds `isOnline` synchronously from `monitor.currentPath`. Consumed by the session-expiry 401 logic (offline ⇒ never logout), `StreamTransportPolicy`'s IPv6 probe scheduling, and SyncPlay gating. **RULE — Bounded timeouts**: every `JellyfinClient` takes `Self.fastFailSessionConfiguration` (request 30s, resource 60s, `waitsForConnectivity=false`); raw PlaybackInfo POST adds `request.timeoutInterval = 20`. `waitsForConnectivity=false` + `NetworkMonitor` detect *offline* instantly, so these only bound the "online but slow server" case — they were 8s/20s/8s but that tore whole screens down to "Serveur injoignable" when a self-hosted server briefly stalled (e.g. libVLC flooding it with AVI range-requests). `AppState.restoreSession` hydrates from keychain immediately, dispatches server calls in detached Task (non-blocking offline launch).
- **Legacy-downloads janitor**: `AppNavigation.purgeLegacyDownloads()` (iOS, root `.task`) deletes the pre-1.0.5 `Application Support/Cinemax/Downloads/` tree + the `downloads.userFlagCache` UserDefaults key left by the removed offline-downloads feature (App Review 5.2.3). Keep until well past 1.0.5 adoption.

## Admin (iOS-only — product decision)

`SettingsCategory.visibleCases(isAdmin:isTVOS:)` short-circuits when `isTVOS`; every `Shared/Screens/Admin/` file `#if os(iOS)`.

- **Gating**: `AppState.isAdministrator` (cached; refreshed on login/reconnect/user switch via `AppState.refreshCurrentUser()`). Server authoritative; client gating is UX. `AppState.currentUser: UserDto?` populated alongside.
- **API surface**: `AdminAPI` slice. Device listing/revocation stays on `AuthAPI` (server returns full fleet to admins by caller identity).
- **Settings routing**: `.administration` (Dashboard + Metadata Manager) and `.advancedAdmin` (Users/Devices/Activity/Playback/Plugins/Catalog/Tasks/Network/Logs/API Keys). Hidden when `!isAdministrator`.
- **Scaffolds** (`Admin/Components/`): `AdminLoadStateContainer`; **RULE — `AdminFormScreen` every admin editor uses explicit save (never auto-save)** — admin changes have blast radius (policy revocations, password resets); sticky footer + `interactiveDismissDisabled(isDirty)` + discard confirmation. `AdminTabBar`, `AdminSectionGroup`. `AdminItemMenu` — shared `Menu` for one `BaseItemDto` (Identifier/Edit/Refresh/Delete); **does NOT host its own `.navigationDestination`** (silently ignored — lazy-container rule), fires `onSelectDestination(_:)`. `DestructiveConfirmSheet` (type-to-confirm) reserved for irreversible ops; reversible destructives use `.confirmationDialog` `.destructive`.
- **Self-protection** (client-side; server enforces too): can't delete/demote/disable yourself; can't revoke current device (`KeychainService.getOrCreateDeviceID()` vs `DeviceInfoDto.id`, "THIS DEVICE" pill).
- **Performance**: Dashboard `async let` fan-out (partial render on single failure). Activity log infinite-scroll (50/page). Tasks live-poll 2s while running, self-cancels.
- **Identify flow** (`Admin/Identify/`): `IdentifyFlowModel` (`@Observable`) — standalone `IdentifyScreen` (form → grid → confirm) + composed in `MetadataEditorViewModel.identify`. Kind-aware (movies: IMDb/TMDb Film/TMDb Coffret; series: IMDb/TMDb/TVDb). Provider IDs under `"Imdb"`/`"Tmdb"`/`"TmdbCollection"`/`"Tvdb"`.
- **Metadata Manager**: five-tab editor (General/Images/Cast/Identify/Actions). Images use `downloadRemoteImage` (server fetches URL, no phone proxy). Delete via `DestructiveConfirmSheet` (title as confirm phrase).
- **ImageType quirk**: CinemaxKit's `ImageType` narrower than `JellyfinAPI.ImageType`. Admin code uses `JellyfinAPI.ImageType` explicitly-qualified; `ImageURLBuilder.imageURLRaw(itemId:imageTypeRaw:)` string-keyed overload renders the wider set without widening `CinemaxKit.ImageType`.
- **RULE — API key security** (`Admin/ApiKeys/`, keys = passwords): masked by default (first 4 + last 4); `.privacySensitive()`; per-row Copy is the only export path (no share sheet); `appState.accessToken` match → `CURRENT SESSION`, revoke hidden; `revokedKeyIds`/`revealedKeyIds` dropped on disappear. **Never log key values or send to analytics.** `revokeApiKey` takes the token itself as identifier (Jellyfin quirk); forget value on return.
- **Poster-card admin overlay**: `LibraryPosterCard` paints ellipsis `AdminItemMenu` bottom-right when `isAdministrator`. Menu and detail-push `NavigationLink` are `ZStack` siblings (not nested).

## MediaDetailScreen

`MediaDetailViewModel` auto-resolves Episode/Season → parent Series (by `seriesID`, loads seasons + episodes) + `getNextUp()`. `selectSeason()` uses a generation counter for stale results; **`load()` and `refreshAfterPlayback()` share a second `loadGeneration` counter** (bump-at-entry, re-check after each await cluster, a superseded pass writes nothing — not even `isLoading`) so an initial load and a dismiss-refresh can't clobber each other's `@Observable` writes. Use `resolvedType` (not initial `itemType`) for layout. **tvOS detail refresh** (`VideoPlayerCoordinator.lastDismissedAt` via `TVDismissDelegate` `onDismiss`; screen `.onChange`): on dismiss the screen runs the TARGETED `refreshAfterPlayback(using:)` — no `isLoading` spinner flash, no similar/seasons re-fetch — re-pulling only the userData-bearing slices (movie: its `getItem`; series: `getItem` + `getNextUp` + the visible season's `getEpisodes`, concurrently — all hit caches `reportPlaybackStopped` invalidated), NOT the full `load()`. iOS reloads via `.task` on pop (unchanged).

- **Resume / next-up** (`actionButtons` → `PlayActionButtonsSection: View, Equatable`): custom `nonisolated static func ==` compares resume + prev/next identity, ignores closure → `.equatable()` short-circuits re-renders. Same pattern on extracted `MediaDetail{Cast,Similar}Section`/`Episode{Card,Row}` (wrap non-Sendable DTO reads in `MainActor.assumeIsolated` — escape hatch #4). tvOS episode list uses `LazyVStack`.
- Movie `playbackPositionTicks > 0` not `isPlayed`: progress bar + remaining + "Lecture" via `PlayLink(startTime:)`. Series: `nextUpEpisode`. **Play from beginning**: when `showResume`, secondary ghost `PlayLink` with `startTime: nil`. `userData.playbackPositionTicks`/`runTimeTicks` are `Int?`; `isPlayed` is `Bool?`.
- **Quality badges** (`MediaQualityBadges.swift`): gated on `@AppStorage("detail.showQualityBadges")`. From `item.mediaSources?.first`. Resolution by height (4K/1080p/720p/SD), HDR (`VideoRangeType` → Dolby Vision/HDR10+/HDR10/HDR), codec, audio format (first-hit: Atmos/TrueHD/DD+/DD/DTS/AAC/FLAC/Opus/MP3), channels. `EmptyView()` when none.
- **Episode nav**: `episodeNavigation(for:)` O(1) from `episodeNavigationMap` (current season) or `nextUpNavigationMap` (cross-season next-up).
- **RULE — `getEpisodes` has a 10s TTL cache** (`JellyfinAPIClient+Library.swift`, keyed `episodes-<seasonId>-<userId>`, mirroring `getItem`) — collapses the burst where Home's nav maps and the detail screen re-fetch the same season within seconds. Because a season's episode list carries per-episode userData (watched marks + resume position), **every userData mutator invalidates the whole `episodes-` prefix**: `markItemPlayed` / `markItemUnplayed` (a series-level toggle cascades to every episode; the mutator only knows the item id, so it can't scope to one season key) and `reportPlaybackStopped` (episode resume position). `refreshVisibleEpisodes` after a played toggle relies on this — without the invalidation it would re-serve the stale cached list and the checkmarks wouldn't update. Any new episode-userData write must add the same `cache.invalidate(prefix: "episodes-")`. **Single-flight**: `getItem` also runs its network fetch through `APICache.coalesce(key:operation:)` (in-flight de-dup at `T == Void`, winner writes the TTL entry, awaiters read it back — `BaseItemDto` never crosses the coalescing boundary), so the playback-start burst's *concurrent* callers (which all miss the TTL before the first response lands) share one round-trip; in-flight entries are removed on success AND failure so a failed fetch never poisons retries. The VLC end-of-series card reuses `fetchChapters`' already-resolved `resolvedSeriesName` instead of re-fetching at playback end.
- **Episode metadata line** (`MediaDetailEpisodeMetadataLine`, own file): shared tvOS/iOS, joined ` • ` — in-progress "Xm remaining" / else runtime / + `premiereDate`. Episode rows/cards paint a `checkmark.circle.fill` watched badge (with VoiceOver watched-state labels); "See more" opens `MediaDetailEpisodeOverviewSheet` (full overview text, iOS `.presentationDetents([.medium, .large])`).
- **Season bulk "mark watched"** (`markSeasonWatchedButton` → `viewModel.markSeasonWatched(seasonId:seriesId:…)`): behind a `confirmationDialog`; `markItemPlayed(seasonId)` cascades to every episode server-side, so all loaded episodes of the visible season are optimistically flipped played, then `refreshVisibleEpisodes` re-syncs, + tier-2 `.cinemaxItemUserDataChanged` + toast (reverts on failure). Button sits trailing beside the season picker (iOS) / in its own focus-section row above the episode list (tvOS); always shown for series with episodes (even single-season). Keys `detail.season.{markWatched,markWatched.message,markedWatched}`.
- **iPad two-column layout**: on iOS regular width (`useTwoColumnLayout` ← `horizontalSizeClass`), below-hero content is a 40/60 `HStack` — primary column (metadata/actions/overview) + secondary (cast/episodes/similar) split via `containerRelativeFrame(.horizontal, count: 5, span:)`. `HomeViewModel.load` fans out a parallel `TaskGroup`: `resumeItems` (limit 20), `latestItems`, `favoriteItems` (movie/series, dateCreated desc, limit 20), `nextUpItems` (`getNextUpEpisodes` limit 20). `heroItem = resumeItems.first ?? latestItems.first`; `hasLoaded` guard prevents tab-remount refetch (`reload` bypasses). **RULE — progressive render**: `isLoading` flips false right after this phase-1 group + `heroItem` (NOT at the end of `load`), so the hero + rails paint while genre rows and the episode-nav maps keep filling in off their own `@Observable` slices (nav maps only gate the in-player prev/next buttons). Episode nav (BOTH the Continue Watching + Next Up rails): `fetchSeasonEpisodes` fetches each referenced season's episode list once, **deduped across BOTH rails** (union of `resumeItems + nextUpItems` seasons — a season referenced by both is fetched a single time; the `getEpisodes` 10s cache backs this up too), then the pure `buildNavigationMap(seasonEpisodes:)` derives each map from the shared dict — `precomputeEpisodeRefs` once/season, O(1) `buildEpisodeNavigation` → `resumeNavigation` / `nextUpNavigation` (`[String:(prev,next,navigator)]`). Both `continueWatchingPlayLink` and `nextUpPlayLink` thread these into `PlayLink` — the player's prev/next episode buttons are gated on non-nil `previousEpisode`/`nextEpisode` (`VLCStreamPresenter` tvOS `controlBar` / iOS `transportRow`), so a rail that omits them shows no episode-nav buttons. Within-season only (mirrors resume; a season-opener Next Up gets `next` but no `prev`). Watching Now: `getActiveSessions(activeWithinSeconds:60)`, drop current user + require `nowPlayingItem`. **RULE — admin-only**: the row, its Settings toggle (`homePageToggleRows`), and the `/Sessions` fetch (`loadActiveSessions`) are all gated on `appState.isAdministrator`. `/Sessions` is meant to be elevated and leaks every user's session to non-admins on some servers (jellyfin#5210) — don't re-enable for non-admins. Configurable layout (`home.show*`, default true; hero never gated). Empty state in `ScrollView` (pull-to-refresh works). **tvOS scroll-to-top on reappearance**: `ScrollViewReader` + `.id("home.top")` attached *directly to `heroSection`* (fallback `Color.clear.frame(height: 0)` only when no hero) + `.onAppear` `scrollTo`. MovieLibraryScreen/SearchScreen/Settings tvOS landing keep the separate zero-height sentinel because their first visible row already sits flush against the safe-area top — see tab-bar pill RULE below.

- **Genre rows are user-configurable + deterministic (de-randomization decision — no more shuffle/random)**: `getGenres` → sorted `localizedCaseInsensitiveCompare` (matches the Settings picker order) → `HomeGenrePreferences.effectiveGenres(available:)` — unconfigured ⇒ deterministic first-6 prefix; configured ⇒ explicit picks re-ordered to canonical order, **no cap**. Fan-out chunked at 6 concurrent (spares self-hosted servers). **Failures become `.failed`** (retry chip); empty success drops the row. Live refresh: `.onChange` of the `home.selectedGenres` `@AppStorage` string → `reloadGenreRows`.
- **Next Up rail** (`home.showNextUp`): global next-unwatched episode per in-progress series — distinct from resume points; cards play from the start (no resume offset). Title = `seriesName`, subtitle `SxxExx - name`. Cards get prev/next episode buttons in the player via `nextUpNavigation` (see Episode nav above).
- **RULE — iOS hero carousel; tvOS keeps the single static hero**: `heroCandidates` = up to 5 distinct from `resumeItems + latestItems`; entries after the first REQUIRE `hasBackdropImage` (crossfade must never flash the fallback). Auto-advance every 8s (`runHeroRotation`, `.easeInOut(0.6)` crossfade, `.task(id:)`-scoped so it cancels on disappear) — **no-op unless `motionEffects` is on AND ≥2 candidates**. Swipe via `simultaneousGesture` `DragGesture` (direction + 50pt threshold). Dots: accent active / `onSurfaceVariant.opacity(0.4)`, centered, `allowsHitTesting(false)` + `accessibilityHidden(true)`; `currentHeroIndex` clamps against a shrinking set. tvOS is deliberately excluded — auto-advancing focusable chrome fights the focus engine / `.focusSection()` machinery.
- **Continue Watching context menu**: long-press (iOS) / long-press-select (tvOS), attached to the PlayLink button not the label (preserves focus). Actions: **Mark as watched** (`markItemPlayed`) and **Remove** (`markItemUnplayed`). **RULE — Jellyfin has no hide-from-resume endpoint** — clearing played state resets the resume position, which is what drops it from `/UserItems/Resume`. - **FavoritesScreen** ("watchlist"): aggregated hearted movies/series — `getItems` `isFavorite`, sorted `.sortName` **ascending**, paginated 40/page via `PaginatedLoader` (deliberately different from the Home Favorites row: dateCreated-desc/20, one-shot). `LazyVGrid` (tvOS 6 cols; iOS `AdaptiveLayout`); reloads on `.cinemaxFavoritesChanged` + `.cinemaxShouldRefreshCatalogue` + tier-2 `.cinemaxItemUserDataChanged` (immediate — an un-heart/un-watch must drop the card while on screen, resetting the paginator to page 0). **Entry is ONLY the Home Favorites row "View All"** (`showViewAll: true` → `favoritesDestination` token → screen-root `navigationDestination(item:)`); not a tab; declares no own `NavigationStack`.

- **RULE — tvOS 26 tab bar pill alignment heuristic**: Liquid Glass reserves a constant 157pt `safeAreaInsets.top` container above the scroll content (verified by `GeometryReader` — identical on every tab), but the active pill aligns to either the **bottom** of that container ("expanded", default) or the **top** ("compact", ~30pt higher visually) based on whether the first scrollable row touches the safe-area top edge. Any leading gap — including the implicit `LazyVStack` `spacing:` between a zero-height sentinel and the first visible row — flips the pill into compact mode, making the menu look like it shifted upward on that tab (and that tab only). Fix pattern: `LazyVStack(spacing: 0)` + explicit `.padding(.bottom, spacing6)` between rows + attach the scroll-anchor `.id(scrollTopID)` directly to the first visible row instead of a separate sentinel. Reference: `HomeScreen.content` (the hero carries `.id(scrollTopID)` and the LazyVStack uses `spacing: 0`). Films/Recherche/Réglages don't need the workaround because their first row (`tvTopBar` / `searchField` / `tvLandingPage`) already sits flush — they have `LazyVStack(spacing: 0)` or use eager VStacks. Diagnosis is non-trivial because content-level padding, focusable proximity, `toolbarBackgroundVisibility`, and body-level modal modifiers all have **no effect** on the pill alignment — only the gap above the first scroll row matters.

## SearchScreen

`SearchViewModel.search(using:)` debounces 400ms → `fetchRanked(...)`. Decomposition: shell + file-private `VoiceSearchButton` (iOS), `SearchResultsGrid`, `SearchResultCard`. iOS mic → `SpeechRecognitionHelper`. **Surprise Me**: `fetchRandomMovie`/`fetchRandomSeries` are separate methods (Swift 6 flags `[BaseItemKind]` built from a parameter as non-Sendable crossing the API actor; literal arrays work).

- **Type filter chips (All / Movies / Series)**: `SearchScope` enum → `viewModel.scope`; shown once a search is under way. `fetchRanked` takes an additive `includeItemTypes: [BaseItemKind]` param threaded into `searchItems` (default `[.movie, .series, .episode]` in the `LibraryAPI` extension so existing callers are unchanged). Changing a chip re-runs the current query (`search(using:)`). tvOS chips use `TVFilterChipButtonStyle`; iOS uses the accent-capsule idiom. Keys `search.filter.{all,movies,series}`.
- **Poster context menus** (`SearchResultCard`): shared `.mediaCardContextMenu(item:)` — long-press / long-press-select watched + favorite toggle (see Media Library RULE).

### Permissive weighted search (`fetchRanked`)

Jellyfin's `searchTerm` is contiguous + punctuation-sensitive — "Mission Impossible" misses "Mission : Impossible". `fetchRanked` (`nonisolated static`, takes `any LibraryAPI` which is `Sendable`) fans out **concurrent** `searchItems` calls for the full phrase **AND each significant word** (`TaskGroup`), unions/dedups by id, then ranks locally. **Stop words** (FR/EN articles/conjunctions, `searchStopWords`) are dropped from per-word fetches + scoring so "the"/"le"/"de" don't match everything. `relevanceScore` tiers (score 0 ⇒ filtered out): **(1)** full normalized query is a contiguous run in title (+bonus exact / prefix) **(2)** every query word present but separated **(3)** only some words present (× fraction). Scored against `name` **and** `originalTitle` (max). `normalizeForMatch` folds diacritics + lowercases + collapses any non-alphanumeric run to a single space.

### Voice search (`SpeechRecognitionHelper`, iOS only) — RULES

- **RULE — every off-main framework callback must be `@Sendable` (nonisolated).** TCC permission callbacks, the `AVAudioEngine` `installTap` block (audio render thread), and the `recognitionTask` result handler all fire off the main actor; a `@MainActor`-isolated closure there traps with `dispatch_assert_queue_fail` / `_swift_task_checkIsolatedSwift`. Fixing only the one in the stack trace leaves the others crashing identically. Permission requests bridge through `nonisolated static func ... async` + `withCheckedContinuation`; the tap captures the request via `nonisolated(unsafe) let` (its `append` is thread-safe); the result handler is `{ @Sendable [weak self] … }` (a `@MainActor` class IS `Sendable`, so weak-self capture is legal) that extracts Sendable values then hops via `Task { @MainActor }`.
- **RULE — empty-transcript guard**: forward a transcript only when non-empty. On `stop()`, `endAudio()` finalizes and can emit a final result with an EMPTY `formattedString`; propagating it wipes `searchText` (and the results) the instant the mic turns off.
- **Silence auto-stop**: buffer recognition never self-reports `isFinal`. A `silenceTimer` (Task) armed at start with a longer `initialTimeout` (5s — server recognition has first-result latency, a short timeout would kill the session before any word lands) then re-armed at `silenceTimeout` (1.5s) on every partial transcript; firing calls `stop()`. `stop()` is idempotent (guards on `audioEngine.isRunning || recognitionRequest != nil`) since it arrives from timer / final result / manual tap.
- **Listening UI**: the trailing toolbar mic button expands into a Liquid-Glass pill (`mic.fill` + `search.listening`, pulsing via `symbolEffect(.pulse, isActive: motionEffects)`) while `isListening` — no separate label, no `.glass` style (iOS 26 auto-wraps toolbar buttons; an explicit capsule would nest).

## Image Patterns (all RULE)

- `ImageURLBuilder` → `/Items/{id}/Images/{type}`. **Backdrop sizing**: use `ImageURLBuilder.screenPixelWidth` — never hardcode `1920`.
- **RULE — always pass the `tag:` cache-buster to `imageURL(...)`/`imageURLRaw(...)`**: without it the image URL is byte-identical across server-side poster/backdrop edits, so Nuke (keyed by URL) serves the stale image forever — a metadata/poster change is invisible until reinstall, and `clearCache()`/"Refresh Catalogue" don't help (they clear the JSON cache, not Nuke's). Pass `tag: item.primaryImageTagValue` for `.primary`, `tag: item.backdropImageTagValue` for `.backdrop` (helpers in `BaseItemDto+Metadata.swift`; backdrop helper mirrors `backdropItemID`'s parent-first resolution), `person.primaryImageTag` for cast. Every new live image call site must thread the tag. Deliberately omitted on `NowPlayingInfoController` (separate `URLSession` cache, re-fetched per playback).
- **Image cache**: `AppNavigation.init()` configures `ImagePipeline.shared` with 500 MB disk + explicit `ImageCache` `costLimit = 256 MB` (Nuke's ~100 MB default evicts mid-render on tvOS — 4K backdrops decode to 4–8 MB each).
- **Prefetch (`PosterPrefetcher`, Shared/ViewModels)**: Home + Library warm card images via Nuke `ImagePrefetcher` after each data load. **RULE — prefetched URLs must be byte-identical to the consuming card's request** (same `maxWidth` AND `tag`) — Nuke keys on the URL, a near-miss warms nothing. Posters = primary/300, continue-watching = backdrop/600. Call `reset()` before a full catalogue refresh so refreshed tags re-prefetch.
- **Backdrop fallback**: `item.backdropItemID` (→ `parentBackdropItemID ?? seriesID ?? id`). **No-backdrop placeholder**: gate on `item.hasBackdropImage` (NOT `backdropItemID` — always non-nil) → `BackdropFallbackView` (centered `film` symbol over `surfaceContainerLow` + accent radial, under `CinemaGradient.heroOverlay`). Wired in `MediaDetailScreen.backdropSection`/`HomeScreen.heroSection`/`LibraryHeroSection`.
- **Always `CinemaLazyImage`** — never `LazyImage` directly. Card containers: `Color.clear` + `.aspectRatio()` + `.frame(maxWidth: .infinity)` + `.overlay { CinemaLazyImage }` + `.clipped()`. **Backdrop (full-bleed ZStack)**: `CinemaLazyImage` must have `.frame(maxWidth:.infinity, maxHeight:.infinity)` — else ZStack sizes from natural 1920px pushing title off-screen. Outer container `LazyVStack(alignment: .leading)`.
- **PosterCard title alignment**: hidden `Text("M\nM").hidden()` placeholder + actual title overlaid top-aligned → uniform row height.

## Build

```bash
# iOS
xcodebuild build -project Cinemax.xcodeproj -scheme Cinemax -destination 'platform=iOS Simulator,name=iPhone 17 Pro'
# tvOS
xcodebuild build -project Cinemax.xcodeproj -scheme CinemaxTV -destination 'platform=tvOS Simulator,name=Apple TV 4K (3rd generation)'
# Regenerate Xcode project
cd Cinemax && xcodegen generate
```

**Build verification gotchas (RULE)**:
- Always pair pipes with `set -o pipefail` (`set -o pipefail; xcodebuild ... | grep ...`) — without it `tail`/`grep` swallow xcodebuild's exit code and a failed build returns 0. Confirm by reading output for `** BUILD SUCCEEDED **` / `** BUILD FAILED **`, not just shell exit.
- Don't run iOS + tvOS builds in parallel against the same DerivedData — they race on `build.db` ("database is locked"). Run serially.

**Tests**: unit tests (swift-testing) live in `Tests/CinemaxKitTests/` but the Xcode target/product is **`CinemaxTests`** (in-app bundle, `TEST_HOST = Cinemax.app`, `@testable import Cinemax`). Run via `xcodebuild test -scheme Cinemax ...`. **RULE — `-only-testing` silently runs 0 swift-testing tests** — run the full suite and grep for `Suite "Name"` / `✔`. **RULE — never add `package: CinemaxKit` to `CinemaxTests` in `project.yml`** (double-links the `Get` dependency → linker errors; the app target already carries it).

**Versioning**: `iOS/Info.plist` + `tvOS/Info.plist` use `$(MARKETING_VERSION)`/`$(CURRENT_PROJECT_VERSION)` — `project.yml` `settings.base` is SSOT. Bump `MARKETING_VERSION` per user-visible release; `CURRENT_PROJECT_VERSION` per archive/upload.
