## 1. Sécurité

### Corrigé
- **S-1 (Faible) — IDs de chemin du proxy loopback devinables.** `CinemaxStreamProxy` exposait `/s/1`, `/s/2`… (compteur monotone) : une app co-résidente pouvait scanner les ports loopback et lire le flux média en cours (jamais le token, qui ne transite que dans l'en-tête `Authorization` côté origine). → IDs remplacés par des **UUID non devinables**, éviction conservée via un tableau d'ordre d'insertion (`targetOrder`).
- **S-2 (Faible) — token `api_key` persisté en clair** dans `DownloadItem.remoteURL` (les `transcodingURL` Jellyfin l'embarquent), sérialisé dans `index.json`. Déjà protégé par `completeUntilFirstUserAuthentication` + exclusion iCloud ; l'URL n'est par ailleurs **jamais relue** pour relancer un téléchargement (re-négociation systématique). → **credentials strippés avant persistance** (`sanitizedRemoteURL`).

### Constaté sain (ne pas ré-auditer)
- Proxy lié à loopback uniquement (`requiredInterfaceType = .loopback`) ; parsing HTTP borné (64 Ko), `/s/<id>` utilisé comme clé de dictionnaire (pas de traversée de chemin), pas d'injection CRLF possible sur `Range`.
- Aucune validation TLS désactivée nulle part ; la sonde IPv6 utilise les options TLS par défaut avec SNI réel.
- `api_key` jamais loggé : tous les sites de log d'URL passent par `redactedURL()` ; aucun `print()` dans le code.
- Écran Admin API Keys conforme (masquage 4+4, `.privacySensitive()`, presse-papier local + expiration 60 s, copie seul export, clé courante non révocable).
- `KeychainService` : `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` partout, pas de sync iCloud.
- ATS : `NSAllowsArbitraryLoadsForMedia` (pas le global) — attendu pour un client Jellyfin auto-hébergé.
- Résidu accepté : les blobs `resume/<id>.resume` d'URLSession embarquent la requête d'origine (en-tête auth) — protégés at-rest, non exportés.

## 2. Performance

### Vérification de non-régression (audits précédents) — TOUT PASSE
Tick arc-en-ciel 10 Hz gaté motion-effects ; `ImagePipeline` configurée une seule fois ; backdrops plafonnés `min(screenPixelWidth, 1920)` ; `totalDiskBytes` mis en cache hors-main ; écritures `DownloadStore` throttlées 5 s ; formatters hissés en `static let` ; snapshot `sortFilter` avant TaskGroup.

### Corrigé
- **P-1 (Moyen) — `HomeViewModel.load` sérialisait 4 phases** (resume/latest → navigation épisodes → genres → sessions actives). Genres + sessions sont désormais lancés en `async let` concurrents avec la phase épisodes (~0,3–1 s gagnée sur serveur distant).
- **P-2 — `CinemaScale.factor` lisait `UserDefaults`** (verrou CFPreferences) à chaque appel de police, des dizaines de fois par évaluation de `body`. → cache statique invalidé par l'unique écrivain (`ThemeManager.uiScale`).
- **P-3 — Pas de garde `hasLoaded` sur Home** : chaque remontage d'onglet (tvOS recrée les hosting controllers au moindre changement de barre) relançait les 5 appels API et re-mélangeait les rangées de genres. → motif `loadInitial`/`reload` aligné sur `MediaLibraryViewModel`.
- **P-4 — fetch hero de bibliothèque sérialisé** après genres+count → troisième `async let`.
- **P-5 — vignettes de chapitres non annulées** au changement d'épisode (jusqu'à N téléchargements 320 px de l'épisode précédent concurrençant l'ouverture du nouveau flux). → tâches suivies dans `chapterThumbTasks`, annulées sur `fetchChapters`/`teardown`.
- **P-6 — `getaddrinfo` synchrone** dans la sonde IPv6 pouvait épingler un thread du pool coopératif pendant tout le timeout résolveur. → déporté sur `DispatchQueue.global` via continuation.
- **P-7 — `CinemaxStreamProxy.stop()` laissait vivre** connexions acceptées et tâches upstream (gaspillage de bande passante après changement de serveur). → teardown déterministe (registre faible de handlers, `cancel()` sur chacun).

### Différé (documenté, non corrigé)
- **P-8 (Faible) — `MetadataLibraryItemsScreen.filteredItems`** filtre à chaque frappe sans debounce (≤ 500 éléments admin — impact mineur).

## 3. Concurrence Swift 6

### Corrigé
- **C-1 (Moyen) — `VLCStreamViewController.navigateToEpisode`** : un Task de navigation reprenant après `teardown()` ressuscitait la lecture (`player.play`), re-signalait une session fantôme et ré-armait un `Timer` répétitif que rien n'invalidait plus (fuite 1 Hz permanente) ; deux pressions Next rapides pouvaient entrelacer l'état de deux épisodes (segments de A sur l'épisode B). → **compteur de génération** (`navGeneration`, même motif que `NowPlayingInfoController`) + re-vérification `!isTearingDown` après les `await`.
- **C-2 — `StreamTransportPolicy.configure(serverURL: nil)`** n'annulait pas la sonde en vol : une sonde de l'ancien serveur pouvait poser `preferProxy = true` après déconnexion. → `probeTask?.cancel()` ajouté ; en bonus, `withTaskCancellationHandler` ferme la socket de sonde dès l'annulation.
- **C-3 — `UpstreamHandler`** installait son `stateUpdateHandler` après le démarrage de la connexion : une coupure VLC dans l'intervalle n'était jamais observée. → vérification de `conn.state` (même queue) avant `task.resume()`.
- **C-4 — `MenuConfigStore.refreshAvailableViews`** sans garde de réentrance (double-tap « Actualiser les bibliothèques » → deux fetches entrelacés). → `guard !isLoadingViews`.
- **C-5 (Info) — `AppState.serverURL` portait un `didSet`** contredisant la RULE maison sans être documenté. → commentaire d'exemption explicite (dérivation de propriété sœur, pas de persistance — le cas visé par la RULE ne s'applique pas).

### Vérifié sain
Discipline de verrou du proxy (`stateLock` exhaustif), analyse de deadlock du backpressure par sémaphore (pas de cycle entre queue déléguée concurrente et `netQueue`), garde de génération NowPlaying, pont `DownloadManager.Adapter`, snapshot TaskGroup de `MediaLibraryViewModel` (correctif de mai toujours en place), `PlaybackReporter`, observer unique du presenter natif.

## 4. Qualité / factorisation

### Corrigé
- **Q-1 (Élevé) — tout le module Admin affichait `error.localizedDescription` brut** (39 sites — `unacceptableStatusCode(401)` visible par l'utilisateur), en violation de la RULE. → paramètre `loc` threadé dans tous les VMs admin, mapping via `loc.userFacingMessage(for:)` ; idem `AdminItemMenu`. Deux chaînes anglaises en dur au passage (`"Invalid URL"`, `"Missing user id"`) → clés localisées.
- **Q-2 (Moyen) — `SettingsScreen.fetchCurrentUser` cassé pour les non-admins** : il appelait `getUsers()` (admin-only, 401 avalé par `try?`) → l'avatar du compte iOS ne s'affichait jamais pour un utilisateur normal. → suppression au profit de `appState.currentUser` (hydraté par `getUserByID` pour tous).
- **Q-3 (Moyen) — divergence du filtre « utilisateurs cachés »** : la grille de changement rapide tvOS affichait les comptes « masqués des écrans de connexion » que `UserSwitchSheet` filtrait. → même filtre appliqué.
- **Q-4 (Moyen) — sleep timer VLC sans gating PiP (iOS)** : le prompt « Toujours là ? » se présentait dans une fenêtre inatteignable quand l'utilisateur regardait en PiP, exactement le bug que la RULE du chemin natif corrige. → pause silencieuse quand l'app n'est pas active (`UIApplication.applicationState`).
- **Q-5 — erreurs téléchargements** : message brut persisté affiché dans la liste (`rowSubtitle`) → statut localisé générique (le brut reste stocké pour diagnostic) ; « Télécharger la saison » avec 100 % d'échecs affichait quand même « téléchargement ajouté » → toast d'erreur quand `queued == 0 && failed > 0`.
- **Q-6 — crash risks** : dernier force-unwrap du code (`parentalRatingOptions.first!`) → fallback localisé ; conversions `Int32(...)` non clampées sur ticks de chapitres → `Int32(clamping:)`.
- **Q-7 — code mort supprimé** : `TrackPickerSheet.swift` (108 lignes, 0 référence), `TVSeriesScreen.swift`, struct `MovieLibraryScreen` (wrapper jamais instancié), modèle `MediaItem`/`MediaType` (échafaudage, seul son propre test l'utilisait), `updateTaskTriggers` (API jamais appelée : protocole + client + mock), `@AppStorage darkModeStorage` jamais lu, **31 clés de localisation orphelines ×2 langues** (vérifiées sans construction dynamique possible ; `app.name` conservée). Tokens `CinemaColor` sans usage marqués « complétude de palette » volontaire.
- **Q-8 — fallback "Chapter N"** non localisé dans `ChapterController` (panneau chapitres tvOS) → clé `player.chapters` réutilisée.

### Différé (assumé)
- **Q-9 — extraction des contrôleurs inline de `VLCStreamPresenter`** (~2000 lignes : skip-segments, sleep timer, fin de série, chapitres dupliqués vis-à-vis des contrôleurs natifs). C'est le report documenté d'Audit_post_vlc : toute extraction change le cycle de vie d'objets dans le code SwiftVLC le plus fragile et **n'est pas validable sans appareil réel** (le simulateur n'a ni HEVC/DV matériel ni PiP). À faire sur une branche dédiée avec passe on-device. Le delta de comportement le plus grave (PiP/sleep) est, lui, corrigé (Q-4).
- **Q-10 — fetchers d'images authentifiés quasi identiques ×3** (`ChapterController`/`VLCStreamPresenter`/`NowPlayingInfoController`) — candidat à un helper partagé lors de Q-9.
- **Q-11 — déplacement de `APICacheTests`/`PlayMethodTests`** vers le package (contrainte `Get` double-link à re-tester sur Mac).

## 8. Revue croisée du diff (vérification des corrections)

Le diff complet des correctifs a été repassé par des relecteurs indépendants sous 5 angles (scan ligne à ligne — interrompu par une limite d'infrastructure, comportements supprimés, traçage inter-fichiers de chaque signature modifiée, réutilisation/simplification, efficacité/altitude). Constats traités en suivi :

- **`HomeViewModel.load` rendu `private` cassait la compilation des tests** (`@testable` n'expose que `internal`) → repassé `internal` avec doc (l'app passe par `loadInitial`/`reload`).
- **La garde de réentrance de `refreshAvailableViews` pouvait droper le refresh d'un changement de compte** et laisser un fetch périmé persister les bibliothèques de l'ancien utilisateur → remplacée par un **compteur de génération** (le plus récent gagne, re-vérification du `userId` après l'`await`) — corrige aussi le « retry silencieusement ignoré ».
- **Le filtre « comptes masqués » pouvait faire disparaître la tuile de l'utilisateur connecté** (admin qui masque son propre compte des écrans de connexion) → exemption `id == currentUserId` ; au passage la règle de visibilité (getUsers admin-only → filtre → fallback public) est **extraite en un seul helper partagé** `AppState.fetchSwitchableUsers` consommé par les deux surfaces (élimine aussi le 401 systématique payé par les non-admins côté tvOS).
- **Garde PiP du sleep timer trop large** (`!= .active` avalait le prompt pendant Centre de contrôle/bannière d'appel) → resserrée à `.background` uniquement, avec commentaire honnête : c'est une approximation (SwiftVLC n'expose pas d'état PiP fiable), pas « la même règle » que le seam natif.
- **« Chapitres 3 » grammaticalement faux** (clé plurielle réutilisée en titre singulier) → clé `player.chapter` ajoutée (FR « Chapitre »), utilisée dans `ChapterController` et la bande de chapitres VLC.
- **Gating motion à la mauvaise altitude** : plutôt que de threader `motionEffectsEnabled` à chaque site `CinemaToggleIndicator(animated:)`, le composant lit désormais l'environnement lui-même (même motif que `TVButtonStyles`) — un site qui omet le paramètre ne peut plus violer la règle.
- **Prédicats « paramètre secret » dupliqués** (`redactedURL` / `sanitizedRemoteURL`) → commentaires croisés KEEP-IN-SYNC de part et d'autre (un refactor partagé inter-modules a été jugé trop risqué sans compilateur).
- Branche iOS de `MediaDetailScreen.buttonFontSize` (18 nu) → `CinemaScale.pt(18)`.

Constats examinés et **non retenus**, avec raison : threading `loc:` par paramètre dans les VMs admin (c'est la convention documentée de CLAUDE.md — « VMs that show errors take `loc:` » ; l'injection à l'init serait un changement de convention à décider hors audit) ; remplacement de `targets`+`targetOrder` par une structure unique et passage des vignettes de chapitres en task group structuré (le code actuel est vérifié correct et borné ; refactor sans compilateur = risque net négatif) ; token partagé pour la taille des labels Play (les tailles iOS divergent par surface — décision design) ; `hasLoaded` armé avant succès du premier load (trade-off assumé, cohérent avec `MediaLibraryViewModel`, chemins de récupération existants — noté pour un éventuel raffinement).
